# 开发环境配置指南

## 概述

本文档介绍如何根据 Docker 环境配置来设置本地开发环境，确保开发环境与生产环境的一致性。

## 环境配置对比

### 1. 后端服务配置

| 配置项 | 开发环境 | Docker环境 | 说明 |
|--------|----------|------------|------|
| 数据库地址 | localhost:3306 | mysql:3306 | Docker使用服务名 |
| Redis地址 | localhost:6379 | redis:6379 | Docker使用服务名 |
| RabbitMQ地址 | localhost:5672 | rabbitmq:5672 | Docker使用服务名 |
| Elasticsearch（可选，暂时禁用） | localhost:9200 | elasticsearch:9200 | Docker使用服务名 |
| 日志级别 | DEBUG | INFO | 开发环境详细日志 |
| SQL日志 | 启用 | 禁用 | 开发时显示SQL |
| 热重载 | 启用 | 禁用 | 开发时自动重启 |

### 2. 前端配置

| 配置项 | 开发环境 | 生产环境 | 说明 |
|--------|----------|----------|------|
| API地址 | http://localhost:8888/api | /api | 开发时直连后端 |
| WebSocket | ws://localhost:8888/ws | ws://localhost:8888/ws | 实时通信 |
| 调试模式 | 启用 | 禁用 | 开发时显示调试信息 |
| Mock数据 | 可选 | 禁用 | 开发时可使用模拟数据 |

## 快速开始

### 1. 启动基础服务（使用Docker）

```bash
# 启动基础服务（MySQL、Redis、RabbitMQ）
docker-compose -f docker-compose.env.yml up -d mysql redis rabbitmq

# 或使用后端容器（基础服务 + 后端）
docker-compose -f docker-compose.dev.yml up -d

# 或者启动完整服务
cd ..
docker-compose up -d
```

### 2. 配置开发环境

#### 后端开发环境

1. **修改 application-dev.yml 中的数据库连接**：
```yaml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/blog_dev?useUnicode=true&characterEncoding=utf8&useSSL=false&serverTimezone=Asia/Shanghai&allowPublicKeyRetrieval=true
    username: root
    password: 123456  # 修改为你的MySQL密码
```

2. **启动后端服务**：
```bash
cd blog-service
./gradlew bootRun --args='--spring.profiles.active=dev'
```

#### 前端开发环境

1. **安装依赖**：
```bash
cd blog-admin
npm install
# 或
pnpm install
```

2. **启动开发服务器**：
```bash
npm run dev
# 或
pnpm dev
```

### 3. 环境变量配置

#### 后端环境变量

创建 `blog-service/.env.dev` 文件：
```properties
# 数据库配置
DB_HOST=localhost
DB_PORT=3306
DB_NAME=blog_dev
DB_USER=root
DB_PASSWORD=123456

# Redis配置
REDIS_HOST=localhost
REDIS_PORT=6379

# RabbitMQ配置
RABBITMQ_HOST=localhost
RABBITMQ_PORT=5672
RABBITMQ_USER=guest
RABBITMQ_PASSWORD=guest

# 其他配置
JWT_SECRET_KEY=dev_secret_key_change_in_production
AMAP_KEY=your_amap_key
MAIL_USERNAME=your_email@qq.com
MAIL_PASSWORD=your_email_password
```

#### 前端环境变量

修改 `blog-admin/.env.local` 文件：
```properties
# API配置
VITE_API_BASE_URL=http://localhost:8888/api
VITE_WS_BASE_URL=ws://localhost:8888/ws

# 开发配置
VITE_DEBUG=true
VITE_MOCK_ENABLED=false
```

## 开发工具配置

### 1. IDE配置

#### IntelliJ IDEA
- 安装 Spring Boot 插件
- 配置 Lombok 插件
- 设置代码格式化规则

#### VS Code
- 安装 Java Extension Pack
- 安装 Spring Boot Extension Pack
- 配置 ESLint 和 Prettier

### 2. 数据库配置

#### 创建开发数据库
```sql
CREATE DATABASE blog_dev CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE USER 'blog_dev'@'localhost' IDENTIFIED BY 'blog_dev_password';
GRANT ALL PRIVILEGES ON blog_dev.* TO 'blog_dev'@'localhost';
FLUSH PRIVILEGES;
```

#### 导入测试数据
```bash
# 如果有测试数据脚本
mysql -u blog_dev -p blog_dev < blog-service/src/main/resources/data-dev.sql
```

### 3. 调试配置

#### 后端调试
在 IntelliJ IDEA 中创建 Spring Boot 运行配置：
- Main class: `com.blog.BlogApplication`
- VM options: `-Dspring.profiles.active=dev`
- Environment variables: 从 `.env.dev` 文件加载

#### 前端调试
在 VS Code 中创建 launch.json：
```json
{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "Launch Chrome",
      "request": "launch",
      "type": "chrome",
      "url": "http://localhost:3000",
      "webRoot": "${workspaceFolder}/blog-admin/src"
    }
  ]
}
```

## 常见问题

### 1. 端口冲突
如果默认端口被占用，可以修改配置：
- 后端：修改 `server.port` 配置
- 前端：修改 `vite.config.ts` 中的 `server.port`

### 2. 跨域问题
开发环境已配置 CORS，如果仍有问题：
1. 检查前端 API 地址配置
2. 确认后端 CORS 配置
3. 使用代理配置

### 3. 数据库连接问题
1. 确认 MySQL 服务已启动
2. 检查用户名密码
3. 确认数据库已创建
4. 检查防火墙设置

### 4. Redis 连接问题
1. 确认 Redis 服务已启动
2. 检查端口配置
3. 确认密码配置（如果设置了密码）

## 部署对比

### 开发环境 vs Docker环境

| 方面 | 开发环境 | Docker环境 |
|------|----------|------------|
| 启动方式 | IDE/命令行 | Docker Compose |
| 配置管理 | 配置文件 | 环境变量 |
| 服务发现 | localhost | 服务名 |
| 数据持久化 | 本地文件 | Docker卷 |
| 日志管理 | 控制台/文件 | 容器日志 |
| 监控 | 开发工具 | Prometheus/Grafana |

## 最佳实践

1. **配置管理**：
   - 使用环境变量管理敏感信息
   - 不要提交 `.env.local` 文件
   - 保持配置文件的一致性

2. **开发流程**：
   - 定期同步 Docker 配置
   - 使用相同的依赖版本
   - 遵循代码规范

3. **测试**：
   - 在 Docker 环境中测试
   - 使用相同的数据库版本
   - 模拟生产环境配置

4. **安全**：
   - 不要在开发环境使用生产密钥
   - 定期更新依赖
   - 使用安全的默认配置
