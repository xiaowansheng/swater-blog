# 部署与发布

本文档聚焦根目录的容器化部署方案，覆盖开发演示、HTTP 直连和 HTTPS 代理三种场景。

## 1. 核心文件

- [docker-compose.yml](../../docker-compose.yml)
- [docker-compose.env.yml](../../docker-compose.env.yml)
- [docker-compose.dev.yml](../../docker-compose.dev.yml)
- [.env.example](../../.env.example)

## 2. Compose 模式

### 2.1 基础依赖模式

```bash
docker compose -f docker-compose.env.yml up -d mysql redis rabbitmq
```

用途：本机联调时只需要基础服务。

### 2.2 开发容器模式

```bash
docker compose -f docker-compose.dev.yml up -d
```

用途：基础服务 + `blog-service` 容器。

### 2.3 完整部署模式

```bash
cp .env.example .env
COMPOSE_PROFILES=http docker compose up -d
```

用途：完整启动 MySQL、Redis、RabbitMQ、`blog-service`、`blog-admin`、`blog-web` 和 HTTP Nginx。

本机没有域名时，HTTP 模式可以直接用 `localhost`：

- 博客前台：`http://localhost`
- 管理后台：`http://localhost:3001`
- 后端 API：`http://localhost:8888`

这种方式不需要修改 hosts。前台必须优先走 `http://localhost`，因为它经过外层 Nginx，`/api` 和 `/uploads` 才会被代理到正确服务。

本地 `.env` 关键配置示例：

```env
COMPOSE_PROFILES=http
BLOG_ADMIN_PUBLISHED_PORT=3001
BLOG_WEB_PUBLISHED_PORT=3002

VITE_API_BASE_URL=/api
VITE_BLOG_URL=http://localhost

BLOG_BOOTSTRAP_ADMIN_USERNAME=admin
BLOG_BOOTSTRAP_ADMIN_PASSWORD=你的初始密码
BLOG_BOOTSTRAP_ADMIN_EMAIL=admin@example.com

NEXT_PUBLIC_API_BASE_URL=/
SERVER_API_BASE_URL=http://blog-service:8888
NEXT_PUBLIC_UPLOAD_RESOURCE_PREFIX=/uploads
NEXT_PUBLIC_SITE_URL=http://localhost
```

### 2.4 HTTPS 代理模式

```bash
cp .env.example .env
COMPOSE_PROFILES=https docker compose up -d
```

用途：启用 `nginx-proxy` + `acme-companion` 自动签发证书。

HTTPS 模式需要让外层代理域名和内层 Nginx 路由域名保持一致：

```env
COMPOSE_PROFILES=https
ALL_DOMAINS=admin.example.com,blog.example.com
NGINX_ADMIN_DOMAIN=admin.example.com
NGINX_WEB_DOMAIN=blog.example.com
ACME_EMAIL=admin@example.com
NGINX_PROXY_CONTAINER=swater-nginx-proxy
```

说明：

- `ALL_DOMAINS` 用逗号分隔，给 `nginx-proxy` 和 ACME 申请证书使用
- `NGINX_ADMIN_DOMAIN` 决定哪个域名进入管理后台
- `NGINX_WEB_DOMAIN` 决定哪个域名进入博客前台
- `NGINX_PROXY_CONTAINER` 默认应等于 [docker-compose.yml](../../docker-compose.yml) 中 `nginx-proxy` 服务的 `container_name`

补充说明：

- 如果仓库位于 WSL 文件系统，比如 `/home/...`，请在 WSL 终端里执行 `docker compose`
- 不要在 Windows 侧对 `\\wsl.localhost\...` 路径直接运行 `docker-compose.exe`，否则文件绑定挂载会报 `includes invalid characters for a local volume name`

## 3. 关键环境变量

### 基础服务

- `DB_NAME` / `DB_USER` / `DB_PASSWORD` / `DB_ROOT_PASSWORD`
- `REDIS_PASSWORD`
- `RABBITMQ_USER` / `RABBITMQ_PASSWORD`

### 业务功能

- `MAIL_HOST` / `MAIL_PORT` / `MAIL_USERNAME` / `MAIL_PASSWORD`
- `AMAP_KEY`
- `JWT_SECRET_KEY`
- `AUTH_EMAIL_SESSION_SECRET`
- `BLOG_BOOTSTRAP_ADMIN_USERNAME`
- `BLOG_BOOTSTRAP_ADMIN_PASSWORD`
- `BLOG_BOOTSTRAP_ADMIN_EMAIL`

### 管理员初始化

- 首次空库启动时，后端会自动检查并创建管理员账号
- 默认用户名来自 `BLOG_BOOTSTRAP_ADMIN_USERNAME`，未设置时为 `admin`
- 初始密码必须显式设置 `BLOG_BOOTSTRAP_ADMIN_PASSWORD`，不提供默认密码
- 管理员用户已存在时不会重置密码
- 生产环境应使用强密码，初始化完成并登录后建议立即修改

### 反向代理

- `COMPOSE_PROFILES=http|https`
- `NGINX_ADMIN_DOMAIN`
- `NGINX_WEB_DOMAIN`
- `ALL_DOMAINS`
- `ACME_EMAIL`
- `NGINX_PROXY_CONTAINER`

### 前端与后端联通

- `VITE_API_BASE_URL=/api`：管理后台浏览器请求走同源 `/api`
- `VITE_BLOG_URL=http://localhost`：管理后台跳转到本地前台
- `NEXT_PUBLIC_API_BASE_URL=/`：前台浏览器请求走当前站点 `/api`
- `SERVER_API_BASE_URL=http://blog-service:8888`：前台 SSR/Server Components 在容器内访问后端
- `NEXT_PUBLIC_SITE_URL=http://localhost`：前台生成站点 URL、sitemap、分享链接时使用

## 4. 默认端口

### 完整部署

- 博客前台：`http://localhost`
- 管理后台：`http://localhost:3001`
- 后端 API：`http://localhost:8888`
- RabbitMQ 管理：`http://localhost:15672`

说明：

- `http://localhost` 是外层 Nginx 入口，推荐作为完整 Docker 模式的前台入口
- `BLOG_WEB_PUBLISHED_PORT` 只是 `blog-web` 容器直连端口，不建议作为前台主入口
- 如果 `80` 端口被占用，需要释放端口，或把 [docker-compose.yml](../../docker-compose.yml) 中 `nginx` 服务的 `"80:80"` 改成其他宿主机端口，例如 `"8080:80"`

### 本地开发

- 管理后台：`http://localhost:3000`
- 博客前台：`http://localhost:3001`
- 后端 API：`http://localhost:8888`
- 监控端点：`http://localhost:8081/actuator`

## 5. 发布前检查

- `.env` 已从 `.env.example` 复制并按环境修改
- `JWT_SECRET_KEY` 和 `AUTH_EMAIL_SESSION_SECRET` 已替换默认值
- 邮件、地图、OSS 等第三方配置已填充
- `COMPOSE_PROFILES` 已按 HTTP / HTTPS 场景设置
- 外部域名与 Nginx 域名变量一致
- 生产环境不暴露不必要端口

