# 文章保存流程（前后端协同）

## 背景
- 目标：避免重复创建、保障自动/手动保存幂等，兼顾离线草稿。
- 核心手段：前端在首存前生成稳定 `articleKey`，所有请求沿用；后端按 `articleKey` 做幂等检查。

## 前端流程（blog-admin）
- 初始化：已有文章从接口拿到 `id/version/articleKey` 注入自动保存；新建时生成 UUID 作为 `articleKey` 并缓存。
- 保存构造：每次保存（自动/手动）都携带同一 `articleKey`，同时带最新 `id`、`version`（新建时 `id` 为空）。
- 队列策略：保存队列只保留最新请求，发送前强制覆盖为当前 `articleKey`、最新 `id`/`version`。
- 本地草稿：离线或失败时以 `articleKey` 优先作为存储键，`id` 未生成也能对应同一篇；保存成功后清理草稿。

## 后端流程（blog-service）
- DTO 支持 `articleKey`；创建时若为空则生成。
- 创建幂等：插入前按 `articleKey` 查询已存在文章，若命中直接返回现有记录（避免重复插入），否则插入新文章。
- 更新：沿用原有按 `id` + 版本号的更新/冲突校验。
- 建议：在数据库为 `article_key` 建唯一索引以提供硬性约束。

## 验证用例
- 新建文章，连续触发自动保存和手动保存：仅一条记录被创建，后续请求为更新。
- 已有文章编辑后多次保存：均命中同一 `id`，版本号递增。
- 断网后输入、恢复网络：本地草稿按 `articleKey` 续传，未生成第二条记录。

## 说说保存流程（blog-admin / blog-service）
- 前端（`src/pages/Talk/Edit.tsx`）：
  - 无自动保存，只有“保存草稿”与“提交发布”两个入口；`submitting/saveStatus` 锁定按钮，防止并发提交。
  - 新建保存/发布均为单次请求；保存草稿固定将状态写为草稿。
  - 创建成功后若是新建，会跳转到该说说编辑页（带上返回的 `id`），后续即走更新。
- 后端（`TalkCommandServiceImpl`）：
  - 创建时生成 `talkKey`（服务端生成），插入后触发事件；未做客户端幂等 key，因前端单次提交已锁定。
  - 更新按 `id` 直接覆盖内容/状态/置顶，无版本号控制。
- 当前风险：
  - 若网络抖动导致前端重复点击绕过按钮锁（极少见），后端没有幂等保护可能多次插入；可选增强是在 `talk_key` 上加唯一索引，并允许前端携带稳定 key（与文章一致）以实现创建幂等。
